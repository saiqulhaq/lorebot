## `src/index.ts` ‚Äî entrypoint

Boot sequence: load env config ‚Üí load behavior config ‚Üí set up the KB clone (+ generated agent) ‚Üí health-check the OpenCode server ‚Üí start the config watcher and KB sync loop ‚Üí open the session store ‚Üí connect to Slack. Handles SIGINT/SIGTERM cleanup.

## `src/config.ts` ‚Äî env config

Parses and validates environment variables (connections and secrets). Collects **all** problems into one `ConfigError` so users fix `.env` in a single pass.

## `src/botconfig.ts` ‚Äî behavior config

Everything for `lorebot.config.json`:

- `validateBotConfig` ‚Äî one-pass validation with per-field problems; unknown sections flagged
- `buildAgentMarkdown` ‚Äî generates the read-only kb agent definition from config
- `diffBotConfigs` ‚Äî field-by-field diff for reload logging
- `watchBotConfig` ‚Äî debounced `fs.watch` hot reload; parse failures keep the previous config
- `matchSensitiveKeyword` ‚Äî case-insensitive refusal matching

## `src/slack.ts` ‚Äî Slack handlers

Two bolt handlers: `app_mention` (new questions) and `message` (thread follow-ups in threads the bot owns). Permission gates (users/channels) silently ignore; sensitive keywords get a configured refusal. The answer flow posts a `_Thinking‚Ä¶_` placeholder and updates it in place ‚Äî every error path updates the placeholder too, never leaving it dangling.

## `src/engine.ts` ‚Äî OpenCode client wrapper

The v2 API flow: `session.prompt` (admission; retries with `delivery: "queue"` on busy) ‚Üí `session.wait` (with timeout via `AbortSignal`) ‚Üí `message.list` ‚Üí extract the newest assistant message after the prompt time. A per-thread promise chain serializes rapid questions. `SessionGone` triggers transparent session recreation.

## `src/store.ts` ‚Äî session store

One sqlite table (`bun:sqlite`, zero deps): `(channel, thread_ts) ‚Üí session_id`. Because OpenCode persists sessions server-side, restoring this mapping after a restart restores full conversation context.

## `src/kb.ts` ‚Äî knowledge base management

Clones the KB repo (or uses `KB_DIR`), runs the interval `git pull --ff-only` loop, and installs the generated agent into `<kb>/.opencode/agents/kb.md` ‚Äî kept out of the clone's git status via `.git/info/exclude`. After each successful pull it re-runs the install, which refreshes the graphify index when the graph changed.

## `src/graphify.ts` ‚Äî knowledge graph index

[Graphify](https://graphify.net) integration. When the KB ships a `graphify-out/` directory:

- `parseGraph` ‚Äî defensive parse of graphify's NetworkX node-link `graph.json` (nodes, links, hyperedges)
- `buildIndexMarkdown` ‚Äî distills the graph into a compact `<kb>/.opencode/graphify-index.md` (topic groups + most-connected entities with related files), capped and never silently truncated
- `buildGraphifyPrompt` ‚Äî the extra prompt section `buildAgentMarkdown` appends so the agent knows how to navigate the graph
- `installGraphifyIndex` ‚Äî detect/regenerate; skips work via a `manifest.json` content hash embedded in the index; graceful no-op when disabled or absent

## `src/format.ts` ‚Äî Slack formatting

Markdown ‚Üí mrkdwn conversion (bold, links, headings; code preserved), citation linkification against `KB_LINK_BASE`, bullet-list truncation, sign-off, and length caps from the formatting config.

## `src/sync/` ‚Äî the graphify sync subsystem

The [sync role](/docs/sync) is a self-contained subsystem in the same package (not a separate npm package): six modules with their own entry point, config section, and env vars, integrated with the core through ~20 lines in `index.ts`. A sync failure can never crash the Slack role.

- **`orchestrator.ts`** ‚Äî the per-app pipeline (clone ‚Üí corpus ‚Üí graphify ‚Üí mirror ‚Üí push/PR/dry-run), terminal statuses, the cross-process lock, the crash-safe `.last-sync.json` success state, and the in-process scheduler
- **`corpus.ts`** ‚Äî corpus assembly with rsync (app layout at root + `kb-docs/`), the git-init shield against parent `.gitignore`s, graphify `extract`/`cluster-only` invocation with timeouts, `BUILD_INFO.json` provenance, and the mirror back into the app clone
- **`github-app.ts`** ‚Äî GitHub App auth from scratch with `node:crypto`: RS256 JWT ‚Üí 1-hour installation token, cached with auto-refresh, `tokenRemote()` URLs passed only as git arguments, secret redaction for all error output, PR open-or-update
- **`git.ts`** ‚Äî process helpers (`run`/`git`/`tryGit`) with timeouts and redaction hooks
- **`env.ts`** ‚Äî `requireSyncEnv()`: collected-error credential validation, enforced only when sync runs
- **`cli.ts`** ‚Äî `bun run sync [--force] [--only a,b] [--dry-run]`, usable from cron without the Slack bot

## `src/logger.ts` ‚Äî logging

Zero-dependency structured logger: levels, component tags, pretty or JSON output. See [Logging](/docs/logging).

## Tests

Each module has a matching `test/*.test.ts` (`bun test`, no mocking framework). The engine's answer-extraction logic is tested against hand-written message fixtures.
